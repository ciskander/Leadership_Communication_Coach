# ClearVoice Taxonomy v3.0 Migration — Claude Code Brief

## What this project is

ClearVoice is an AI-powered leadership communication coaching tool. It analyzes meeting transcripts and provides coaching feedback based on a structured taxonomy of communication patterns. The system works as follows:

1. A meeting transcript is fed to an LLM along with a system prompt.
2. The system prompt includes (or is built from) a pattern taxonomy file that defines what communication behaviors to evaluate.
3. The LLM returns a structured JSON response conforming to a defined schema.
4. The backend parses that JSON and hands it to the frontend for display.

We have just completed a major redesign of the pattern taxonomy (v2.1 → v3.0). The new taxonomy file is checked in at `clearvoice_pattern_taxonomy_v3_0.txt`. The task is to update the JSON schema, backend, and frontend to accommodate the new taxonomy.

---

## What changed in the taxonomy and why

### Motivation
The v2.1 taxonomy had 10 patterns. Several overlapped significantly (turn_allocation, facilitative_inclusion, and conversational_balance all measured aspects of the same skill). Some patterns were binary (agenda_clarity scored 0% or 100%). One pattern (listener_response_quality) was acknowledged as unreliable for transcript-based LLM detection. And there were coverage gaps — no patterns for conflict handling, communication clarity, focus management, or structured feedback delivery.

### Structural changes (10 patterns → 9 patterns, 4 clusters)

**Cluster A — Meeting Structure & Direction:**
1. `purposeful_framing` — MERGED from `agenda_clarity` + `objective_signaling`. The meeting opening is now just the first scored opportunity among several transitions, eliminating the binary scoring problem.
2. `focus_management` — NEW. Measures whether the leader manages drift and tangents. Reactive counterpart to purposeful_framing's proactive behavior.

**Cluster B — Participation & Interpersonal Dynamics:**
3. `participation_management` — MERGED from `turn_allocation` + `facilitative_inclusion` + `conversational_balance`. Tiered scoring replaces three overlapping patterns. The `balance_assessment` qualitative annotation (over_indexed / balanced / under_indexed / unclear) is now an optional field on this pattern, not a standalone measurement type.
4. `disagreement_navigation` — NEW. Measures how the leader handles pushback, conflict, and disagreement.

**Cluster C — Decisions & Accountability:**
5. `resolution_and_alignment` — MERGED from `decision_closure` + `summary_checkback`. Scores on two elements: naming the resolution + confirming alignment.
6. `assignment_clarity` — UPGRADED from `owner_timeframe_specification`. Now uses complexity-tiered scoring: simple assignments scored on 3 elements, complex assignments scored on 5 elements.

**Cluster D — Communication Quality:**
7. `question_quality` — REFINED. No longer penalizes strategically open questions. Scores binary: purposeful (1.0) vs. not purposeful (0.0).
8. `communication_clarity` — NEW. Measures BLUF, relevance, proportionality, concreteness of the speaker's own contributions.
9. `feedback_quality` — NEW. Uses the SBI-RC framework (Situation, Behavior, Impact, Request, Check-in). Scoped primarily to 1:1 meetings.

**Retired:** `listener_response_quality` (unreliable detection), `conversational_balance` (absorbed into `participation_management`).

### Scoring model changes

The old taxonomy used binary numerator/denominator/ratio scoring for all patterns except conversational_balance. The new taxonomy uses five scoring types:

1. **Dual-element** (purposeful_framing, resolution_and_alignment): Two named elements per opportunity. Both present = 1.0, one present = 0.5, neither = 0.0. Output includes `element_a_count`, `element_b_count`, `opportunity_count`, `score`.

2. **Tiered rubric** (participation_management, disagreement_navigation, communication_clarity, focus_management): Each opportunity scored 0.0, 0.5, or 1.0 per pattern-specific rubric. Output includes `opportunity_count`, `score`.

3. **Binary** (question_quality): Each opportunity scored 1.0 or 0.0. Output includes `opportunity_count`, `score`. (This is closest to the old model but uses `score` not `ratio`.)

4. **Complexity-tiered** (assignment_clarity): LLM classifies each assignment as simple (3 elements, scored in increments of 0.25/0.5/1.0) or complex (5 elements, scored in increments of 0.25/0.5/0.75/1.0). Output includes `simple_count`, `complex_count`, `opportunity_count`, `score`.

5. **Multi-element** (feedback_quality): 5 sequential SBI-RC elements per feedback instance. Score = elements_present / 5 per instance. Output includes per-instance element presence, `opportunity_count`, `score`.

### Key design decision: scoring_type as discriminator

We decided that the schema should use a `scoring_type` field as the discriminator for the PatternMeasurement union, NOT pattern_id. This means downstream code routes parsing logic based on what kind of scoring is being used (dual_element, tiered_rubric, binary, complexity_tiered, multi_element), not which specific pattern it is. This is simpler, more extensible, and matches how the data will actually be consumed.

## What needs to change

### 1. JSON Schema (`mvp_v0_2_1.json` → next version)

**PatternId enum:** Replace all 10 old IDs with the 9 new ones.

**PatternMeasurement types — full redesign:**
- Remove `ConversationalBalanceMeasurement` as a standalone type. The `balance_assessment` field becomes an optional annotation on any pattern measurement (in practice, only used by `participation_management`).
- Remove all hardcoded `conversational_balance` references throughout the schema (they appear in PatternMeasurementEvaluableNumeric, PatternMeasurementEvaluableMedian, PatternMeasurementInsufficientSignal, PatternMeasurementNotEvaluable).
- Replace the binary `numerator`/`denominator`/`ratio` model with a `scoring_type`-discriminated union. Each scoring type defines which fields are required.
- Rename `ratio` to `score` throughout — this better describes what the field contains for non-binary scoring types.
- The `OpportunityEvent.success` field (currently enum ["yes", "no", "na"]) needs to become a numeric score (0.0, 0.25, 0.5, 0.75, 1.0) or equivalent, since opportunities can now have partial scores.
- Add new fields per scoring type:
  - Dual-element: `element_a_count`, `element_b_count`
  - Complexity-tiered: `simple_count`, `complex_count`
  - Multi-element: per-instance element presence tracking
- Add `scoring_type` field to measurement base with enum: `["dual_element", "tiered_rubric", "binary", "complexity_tiered", "multi_element"]`
- Add optional `cluster_id` field with enum: `["meeting_structure", "participation_dynamics", "decisions_accountability", "communication_quality"]` — useful for frontend grouping.
- `balance_assessment` becomes an optional field on PatternMeasurementBase (not a separate measurement type).

**OpportunityReasonCode enum:** Review and update. Some codes are still relevant (explicit_decision_point, explicit_assignment, agenda_transition). Others may need to be added for new patterns (drift_episode, disagreement_moment, feedback_instance, clarity_assessment, etc.). Consider whether this should become a free-text field with a naming convention instead of a fixed enum.

**The `tier` field:** Investigate what this does in the current system and whether it's still needed. If it's a pattern importance tier, the cluster structure may replace it.

**Version constants:** Update `taxonomy_version` and `schema_version`.

### 2. Backend — Taxonomy parsing (`backend/core/prompt_builder.py` and related)

**Parsing compatibility — largely OK.** The taxonomy file format was designed to work with the existing parsers:
- Section delimiters (`### BEGIN:PATTERN:{id} ###` / `### END:PATTERN:{id} ###`) are preserved.
- `### BEGIN:CORE_RULES ###` and `### END:GENERAL_DETECTION_GUIDANCE ###` are preserved.
- New `### BEGIN:CLUSTER_X ###` / `### END:CLUSTER_X ###` markers are added — these won't break existing parsing but are available for future use.
- Experiment guidance fields use the original names: `What good looks like:`, `Common failure mode:`, `Experiment focus:`, `What it measures:`. The parser's `_extract_field()` and `build_experiment_taxonomy_block()` will work without changes.

**What to check:**
- Any hardcoded pattern ID lists (the debug logging in `workers.py` that validates pattern IDs via regex should work dynamically, but check for any hardcoded expected-ID sets).
- Any code that handles `conversational_balance` as a special case.
- Any code that expects exactly 10 patterns.
- The `allowed pattern_id only:` list in the taxonomy core rules has already been updated.

### 3. Backend — Response parsing and validation

- Update JSON schema validation to the new schema.
- Update any code that reads `numerator`/`denominator`/`ratio` to read `score` and scoring-type-specific fields.
- Update any code that handles `conversational_balance` specially.
- Update any code that maps pattern IDs to display names, categories, or orderings.

### 4. Frontend

- Update pattern ID → display name mappings.
- Update any pattern-specific rendering logic (e.g., conversational_balance had a unique rendering since it was qualitative-only).
- `participation_management` now has both a numeric score AND a qualitative `balance_assessment` — the frontend should render both.
- Consider adding cluster-level grouping to the coaching report display. The clusters are: Meeting Structure & Direction, Participation & Interpersonal Dynamics, Decisions & Accountability, Communication Quality. Each has exactly 2 patterns (except Communication Quality which has 3).
- Update any hardcoded pattern counts (the system now produces 9 patterns instead of 10).

### 5. System prompts

- The system prompts that receive the full taxonomy (`system_prompt_v0_2_1.txt`, `system_prompt_baseline_pack_v0_2_1.txt`) will automatically get the new taxonomy content via `build_developer_message()`. But the prompt text itself may reference old pattern names, the old scoring model, or the old schema structure. Review and update.
- The experiment suggestion prompt (`system_prompt_next_experiment_v0_2_1.txt`) receives extracted experiment guidance via `build_experiment_taxonomy_block()`. This will work automatically with the new taxonomy, but the prompt text may need updating if it references old patterns.
- All prompts that describe the expected JSON output format need to be updated to match the new schema.

---

## Recommended sequence

1. **Start with the schema** — this is the contract between the LLM output and everything downstream. Get this right first.
2. **Update system prompts** — make sure the LLM is asked to produce output matching the new schema.
3. **Update backend parsing/validation** — make sure the backend can consume the new output format.
4. **Update frontend** — make sure the UI can render the new structure.
5. **Test end-to-end** with a real transcript to verify the full pipeline.

## Things to watch out for

- **The `balance_assessment` migration:** This goes from being a standalone measurement type (ConversationalBalanceMeasurement) to an optional annotation on participation_management. Every place in the codebase that handles the old ConversationalBalanceMeasurement needs updating.
- **The binary → partial scoring shift:** Any downstream logic that treats pattern scores as binary (pass/fail, success/failure) needs to be updated for the 0.25/0.5/0.75 partial scores.
- **The `numerator` → `score` rename:** This touches the schema, the backend parsing, and potentially the frontend display. Search the codebase for all references to `numerator`, `denominator`, and `ratio`.
- **Meeting type scoping for feedback_quality:** This pattern is `not_evaluable` in standard team meetings. The backend/frontend should handle this gracefully (it's the same as the existing not_evaluable handling, but it's a new pattern that will frequently be in that state).
- **Opportunity events now carry partial scores:** The old OpportunityEvent.success was yes/no/na. The new model needs to represent 0.0, 0.25, 0.5, 0.75, 1.0. Make sure the event-level detail and the pattern-level aggregate are consistent.
